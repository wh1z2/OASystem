## 1. 系统概述

### 1.1 项目背景

工单流程自动化系统（OA Approval System）是一款面向中小型企业的办公自动化解决方案，旨在通过数字化手段优化企业内部审批流程，提高工作效率，降低管理成本。系统采用现代化的技术架构，支持灵活的流程配置和表单定制，满足企业多样化的业务需求。

### 1.2 系统目标

| 目标维度 | 具体目标描述 |
|---------|-------------|
| **效率提升** | 将传统纸质审批流程数字化，平均审批周期缩短50%以上 |
| **流程透明** | 实现审批全流程可视化，申请人可实时跟踪审批进度 |
| **灵活配置** | 支持自定义审批表单和流程，无需编程即可适配新业务场景 |
| **数据安全** | 建立完善的权限控制体系，确保敏感数据访问安全 |
| **易于集成** | 提供标准API接口，支持与现有企业系统对接 |

### 1.3 系统边界

```

├─────────────────────────────────────────────────────────────────┤
```

### 1.4 术语定义

| 术语 | 英文 | 定义 |
|-----|------|------|
| 工单 | Ticket | 需要审批的业务申请单，包含申请内容和审批流程 |
| 流程实例 | Process Instance | 某个具体工单在审批流程中的执行实例 |
| 状态机 | State Machine | 描述工单状态转换规则的有限状态自动机 |
| 审批节点 | Approval Node | 流程中需要审批人处理的关键环节 |
| 表单模板 | Form Template | 预定义的表单字段集合，用于规范申请内容 |

### 2.2 模块详细说明

#### 2.2.1 用户认证模块

**功能描述**: 提供系统安全访问控制，包括用户登录、登出、会话管理等功能。

**核心功能点**:
- 用户名/密码登录认证
- JWT Token会话管理
- 登录状态持久化（本地存储）
- 权限路由守卫

**接口清单**:
| 接口 | 方法 | 描述 |
|-----|------|------|
| `/api/auth/login` | POST | 用户登录 |
| `/api/auth/logout` | POST | 用户登出 |
| `/api/auth/refresh` | POST | Token刷新 |

#### 2.2.2 仪表盘模块

**功能描述**: 提供系统概览视图，展示关键业务指标和待办事项。

**核心功能点**:
- 统计卡片（待办、已通过、已拒绝、本月申请）
- 待办事项列表（前5条）
- 快捷操作入口
- 审批类型分布统计

**数据指标**:
```javascript
{
  processing: 0,   // 审批中数量（待办）
  approved: 0,     // 已通过数量
  returned: 0,     // 已打回数量
  total: 0         // 本月申请总数
}
```

## 4. 业务流程设计

### 4.1 核心业务流程

#### 4.1.1 工单审批流程（基于COLA状态机）

**状态流转规则说明**:

| 路径 | 起始状态 | 触发操作 | 目标状态 | 权限要求 |
|-----|---------|---------|---------|---------|
| 路径一 | 草稿 | 提交 | 审批中 | 申请人本人 |
| 路径一 | 审批中 | 审批同意 | 已通过 | 当前审批人 |
| 路径一 | 已通过 | 重新编辑 | 草稿 | 申请人本人 |
| 路径二 | 草稿 | 提交 | 审批中 | 申请人本人 |
| 路径二 | 审批中 | 审批不同意 | 已打回 | 当前审批人 |
| 路径二 | 已打回 | 重新编辑 | 草稿 | 申请人本人 |

```
┌──────────────────────────────────────────────────────────────────────┐
│                      工单审批状态机（v1.1）                           │
│                                                                      │
│   ┌──────────┐                                                      │
│   │  DRAFT   │ ←─────────────────────────────────────────┐          │
│   │  (草稿)  │                                           │          │
│   └────┬─────┘                                           │          │
│        │ SUBMIT（提交）                                   │          │
│        ▼                                                 │          │
│   ┌──────────┐                                           │          │
│   │PROCESSING│                                           │          │
│   │ (审批中) │                                           │          │
│   └────┬─────┘                                           │          │
│        │                                                 │          │
│        ├──────────────────┬──────────────────┐           │          │
│        │                  │                  │           │          │
│        ▼                  ▼                  │           │          │
│   ┌──────────┐      ┌──────────┐            │           │          │
│   │ APPROVED │      │ RETURNED │            │           │          │
│   │ (已通过) │      │ (已打回) │            │           │          │
│   └────┬─────┘      └────┬─────┘            │           │          │
│        │                 │                  │           │          │
│        │ REDEDIT         │ REDEDIT          │           │          │
│        │ （重新编辑）     │ （重新编辑）      │           │          │
│        └─────────────────┴──────────────────┘           │          │
│                          │                              │          │
│                          └──────────────────────────────┘          │
│                                                                      │
│   状态变更触发条件验证：                                              │
│   1. SUBMIT: 表单数据完整、必填项已填写                              │
│   2. APPROVE: 当前用户为指定的审批人、审批意见非空                    │
│   3. REJECT: 当前用户为指定的审批人、审批意见非空（必填）             │
│   4. REDEDIT: 当前用户为工单创建人、工单状态为APPROVED或RETURNED     │
└──────────────────────────────────────────────────────────────────────┘
```

#### 4.1.2 审批事件定义

| 事件名称 | 事件代码 | 代码值 | 触发条件 | 业务逻辑 | 权限校验 |
|---------|---------|--------|---------|---------|---------|
| 提交申请 | SUBMIT | 0 | 用户提交工单 | 工单状态变为1(PROCESSING)，记录提交历史 | 申请人本人 |
| 审批同意 | APPROVE | 1 | 审批人点击同意 | 工单状态变为2(APPROVED)，流程结束 | 当前审批人 |
| 审批不同意 | REJECT | 2 | 审批人点击不同意 | 工单状态变为3(RETURNED)，需重新编辑 | 当前审批人 |
| 重新编辑 | REEDIT | 3 | 申请人点击重新编辑 | 工单状态变为0(DRAFT)，保留历史记录 | 申请人本人 |
| 撤销申请 | REVOKE | 4 | 申请人撤销（审批中） | 工单状态变为0(DRAFT)，仅在PROCESSING时可撤销 | 申请人本人 |

#### 4.1.3 状态变更历史记录功能

**记录内容**:
| 字段 | 说明 |
|-----|------|
| 操作人 | 执行状态变更的用户ID及姓名 |
| 操作类型 | SUBMIT/APPROVE/REJECT/REEDIT/REVOKE |
| 操作时间 | 状态变更的精确时间戳 |
| 原状态 | 变更前的工单状态 |
| 新状态 | 变更后的工单状态 |
| 审批意见 | 审批人填写的意见（如适用） |
| 附件 | 相关附件链接（如适用） |

**历史记录查询**:
- 在审批详情页面展示完整的状态变更时间线
- 支持按操作类型筛选历史记录
- 历史记录不可删除、不可修改

#### 4.1.3 流程条件判断

```java
// 伪代码：审批流转条件判断
if (工单金额 > 10000元) {
    需要管理员审批 = true;
} else if (工单类型 == "采购申请") {
    需要管理员审批 = true;
} else {
    经理审批通过即结束;
}

if (申请人部门 != 经理部门) {
    转交对应部门经理审批;
}
```

### 6.3 状态定义

#### 6.3.1 工单状态 (approval_status)

| 状态码 | 状态名称 | 说明 | 可执行操作 |
|--------|---------|------|-----------|
| 0 | DRAFT | 草稿 | 未提交的工单，可编辑修改 | 提交(1) |
| 1 | PROCESSING | 审批中 | 已提交，正在等待审批人处理 | 审批同意(2)、审批不同意(3)、撤销(4) |
| 2 | APPROVED | 已通过 | 审批通过，流程结束 | 重新编辑(5) |
| 3 | RETURNED | 已打回 | 审批不同意，需修改后重新提交 | 重新编辑(5) |
| 4 | REVOKED | 已撤销 | 申请人主动撤销（可选状态） | - |

**状态流转矩阵**:
```
              目标状态
            0(DRAFT)  1(PROCESSING)  2(APPROVED)  3(RETURNED)
         ┌─────────┬─────────────┬───────────┬───────────┐
0(DRAFT) │    -    │   SUBMIT(1) │     -     │     -     │
1(PROCESS)│REVOKE(4)│      -      │APPROVE(2) │ REJECT(3) │
2(APPROVED)│REEDIT(5)│     -       │     -     │     -     │
3(RETURNED)│REEDIT(5)│     -       │     -     │     -     │
         └─────────┴─────────────┴───────────┴───────────┘
```

#### 6.3.2 优先级 (priority)

| 状态码 | 编码 | 名称 | 说明 |
|--------|-----|------|------|
| 0 | LOW | 低 | 普通优先级 |
| 1 | NORMAL | 普通 | 标准优先级 |
| 2 | HIGH | 紧急 | 高优先级，需要优先处理 |

#### 6.3.3 审批类型 (approval_type)

| 状态码 | 编码 | 名称 | 说明 |
|--------|-----|------|------|
| 1 | LEAVE | 请假申请 | 员工请假 |
| 2 | EXPENSE | 报销申请 | 费用报销 |
| 3 | PURCHASE | 采购申请 | 物品采购 |
| 4 | OVERTIME | 加班申请 | 加班申请 |
| 5 | TRAVEL | 出差申请 | 出差申请 |

### 7.1 性能需求

| 指标 | 目标值 | 测量方法 |
|-----|--------|---------|
| 页面首屏加载时间 | < 2秒 | Lighthouse性能评分 |

| 数据库查询时间 | < 100ms | 慢查询日志 |

**优化策略**:
- 前端资源gzip压缩，启用CDN加速
- 数据库索引优化，热点数据缓存
- 分页查询，避免大数据量全表扫描
- 图片懒加载，组件按需加载

### 7.2 安全需求

| 安全维度 | 需求描述 | 实现方案 |
|---------|---------|---------|
| 认证安全 | 防止暴力破解 | 登录失败次数限制(5次/15分钟) |
| 会话安全 | 防止会话劫持 | JWT Token + HTTPS传输 |
| 密码安全 | 防止密码泄露 | BCrypt加密存储，强制复杂度 |
| 访问控制 | 防止越权访问 | RBAC权限校验，后端二次验证 |
| 输入安全 | 防止注入攻击 | 参数化查询，XSS过滤 |
| 数据安全 | 敏感数据保护 | 敏感字段加密存储 |

**安全规范**:
- 所有API接口必须携带有效Token
- 敏感操作需记录审计日志
- 用户密码最小长度8位，需包含字母和数字
- 生产环境强制HTTPS
