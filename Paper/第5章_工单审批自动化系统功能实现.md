# 5 工单审批自动化系统功能实现

## 5.1 开发环境搭建

本项目采用前后端分离的架构模式，前端与后端分别独立部署和运行。开发环境的搭建涵盖操作系统、开发工具、数据库、运行时环境以及各类框架与中间件的配置。

后端开发环境以 Java 生态为主。操作系统采用 Windows 11，开发工具使用 IntelliJ IDEA，构建工具采用 Maven 3.9 版本，项目对象模型文件 `pom.xml` 中定义了完整的依赖树。后端核心框架为 Spring Boot 3.2.3，基于 Java 17 运行，内嵌 Tomcat 容器，服务端口配置为 8080，上下文路径为 `/api`。持久层框架选用 MyBatis-Plus 3.5.10.1 搭配 MySQL 8.0 数据库，连接池使用 HikariCP，最小空闲连接数为 5，最大连接数为 20。状态机引擎采用阿里巴巴 COLA 组件中的 `cola-component-statemachine` 5.0.0 版本，用于驱动审批工单的状态流转。安全与认证方面引入 Spring Security 和 JJWT 0.12.3，密码加密采用 BCrypt 算法。此外，项目还依赖 FastJSON2 2.0.46 用于 JSON 序列化，Lombok 用于简化实体类代码。数据库迁移脚本存放于 `src/main/resources/db/migration/` 目录下，采用手动执行方式管理 schema 变更。

前端开发环境基于 Node.js 生态。构建工具使用 Vite 5.2.0，开发服务器运行于 3000 端口，启动时自动打开浏览器。前端框架采用 Vue 3.4.21，配合 Vue Router 4.3.0 实现单页应用路由，状态管理使用 Pinia 2.1.7。样式方案采用 Tailwind CSS 3.4.3 搭配 PostCSS 与 Autoprefixer，在 `tailwind.config.js` 中自定义了主色、成功色、警告色和危险色的调色板。HTTP 请求库使用 Axios 1.6.8，图标库采用 `@heroicons/vue` 2.1.1。前端通过 Vite 的代理配置将 `/api` 路径转发到后端 8080 端口，实现开发环境下的跨域联调。

环境配置要点方面，后端 `application.yml` 中配置了 JWT 密钥、Token 有效期（Access Token 30 分钟、Refresh Token 7 天）以及 Jackson 的日期格式与时区。`application-dev.yml` 中配置了开发环境的数据库连接参数，并开启 MyBatis-Plus 的 SQL 日志输出以便调试。前端 `api/config.js` 中创建了 Axios 实例，统一设置 baseURL 为 `/api`，请求超时时间为 10 秒，并配置了请求与响应拦截器用于 Token 的自动附加与刷新的处理。

## 5.2 用户管理功能实现

### 5.2.1 用户账号管理

用户账号管理功能面向系统管理员，提供对企业内部用户账号的全生命周期管理能力。前端管理界面位于 `UserManage.vue` 页面，采用表格形式展示用户列表，表格列涵盖用户头像与基本信息、用户名、邮箱、所属部门、角色以及账号状态。页面顶部提供搜索框和角色筛选下拉框，支持按用户名、姓名、邮箱进行关键词检索，以及按系统管理员、部门经理、普通员工三种角色进行过滤。管理员点击"添加用户"按钮后，弹出模态框填写姓名、用户名、邮箱、手机号、部门和角色等信息，其中用户名具有唯一性约束。对于已有用户，管理员可点击"编辑"按钮修改其基本信息，或点击"删除"按钮移除账号；系统特别保护了管理员账号 `admin`，禁止对其进行删除操作。

后端用户管理接口由 `UserController` 提供，基础路径为 `/users`，包含查询、创建、更新、删除四类操作。`UserServiceImpl` 在创建用户时使用 `PasswordEncoder` 对明文密码进行 BCrypt 加密，并通过查询用户名是否已存在来保证唯一性约束。删除操作中对系统管理员账号 `admin` 进行了硬性保护，核心实现如下：

```java
@Override
public Long createUser(UserCreateRequest request) {
    User existUser = userMapper.selectOne(
            new LambdaQueryWrapper<User>().eq(User::getUsername, request.getUsername()));
    if (existUser != null) {
        throw new BusinessException("用户名已存在");
    }
    User user = new User();
    BeanUtils.copyProperties(request, user);
    user.setPassword(passwordEncoder.encode(request.getPassword()));
    userMapper.insert(user);
    return user.getId();
}
```

上述接口均使用 `@PreAuthorize` 注解限制仅拥有 `user_manage` 或 `all` 权限的用户可访问，结合 Spring Security 的方法级安全机制实现接口权限闭环。

### 5.2.2 个人信息管理

个人信息管理功能面向普通登录用户，允许其查看和修改自己的个人资料及密码。前端入口位于 `Profile.vue` 个人中心页面，用户可以修改姓名、邮箱、手机号等基本信息，也可以单独修改登录密码。密码修改界面要求用户输入旧密码和新密码，前端仅做非空校验，具体的密码正确性校验由后端完成。

后端提供 `POST /users/password` 修改密码接口和 `POST /users/profile` 更新个人资料接口，均要求用户已登录。修改密码的核心逻辑通过 `PasswordEncoder.matches` 比对旧密码的 BCrypt 哈希值，若匹配则将新密码加密后落库：

```java
@Override
public Boolean changePassword(Long userId, PasswordChangeRequest request) {
    User user = userMapper.selectById(userId);
    if (user == null) {
        throw new BusinessException("用户不存在");
    }
    if (!passwordEncoder.matches(request.getOldPassword(), user.getPassword())) {
        throw new BusinessException("旧密码错误");
    }
    user.setPassword(passwordEncoder.encode(request.getNewPassword()));
    userMapper.updateById(user);
    return true;
}
```

接口通过 `@PreAuthorize("isAuthenticated()")` 确保只有已登录用户可调用，当前用户 ID 从 `SecurityContextHolder` 中的 `Authentication` 对象提取，以避免越权操作他人账户。

### 5.2.3 用户数据查询

用户数据查询功能主要服务于用户列表页的展示和检索需求。前端 `UserManage.vue` 页面在挂载时调用 `userStore.fetchUsers()` 加载用户数据，并在用户输入搜索关键词或切换角色筛选条件时，通过计算属性 `filteredUsers` 对本地数据进行前端过滤。这种设计在当前数据量较小的情况下能够提供即时响应的搜索体验。

后端 `UserServiceImpl.listUsers` 方法实现服务端的分页与多条件筛选，使用 MyBatis-Plus 的 `Page` 进行物理分页，并通过 `LambdaQueryWrapper` 动态拼接查询条件。其中关键词检索使用 `wrapper.and` 包裹一个 OR 子句，同时匹配用户名、姓名和邮箱三个字段：

```java
LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>();
if (StringUtils.hasText(query.getKeyword())) {
    wrapper.and(w -> w.like(User::getUsername, query.getKeyword())
            .or().like(User::getName, query.getKeyword())
            .or().like(User::getEmail, query.getKeyword()));
}
if (query.getRoleId() != null) wrapper.eq(User::getRoleId, query.getRoleId());
if (query.getDeptId() != null) wrapper.eq(User::getDeptId, query.getDeptId());
wrapper.orderByDesc(User::getCreateTime);
Page<User> result = userMapper.selectPage(page, wrapper);
```

查询结果按创建时间降序排列，对于查询到的每个 `User` 实体，`convertToUserInfo` 方法会进一步关联查询 `Role` 与 `Department` 表，填充角色名称和部门名称，最终封装为 `PageResult` 返回。这种设计既支持灵活的多条件检索，又通过服务端分页避免了大数据量下的性能问题。

### 5.2.4 用户登录

用户登录功能采用账号密码登录方式实现。前端登录页面 `Login.vue` 提供用户名和密码输入框，密码输入支持显隐切换，前端仅做非空校验。用户在输入用户名和密码后点击登录，前端调用 `authStore.login` 方法向 `POST /auth/login` 接口发送请求。后端 `AuthServiceImpl` 利用 Spring Security 的 `AuthenticationManager` 完成账号密码校验，通过后生成 Access Token 与 Refresh Token，并将用户基本信息一并封装进 `LoginResponse` 返回，核心实现如下：

```java
Authentication authentication = authenticationManager.authenticate(
        new UsernamePasswordAuthenticationToken(
                request.getUsername(), request.getPassword()));
SecurityContextHolder.getContext().setAuthentication(authentication);
UserDetailsImpl userDetails = (UserDetailsImpl) authentication.getPrincipal();

String accessToken = jwtTokenUtil.generateToken(userDetails.getId(), userDetails.getUsername());
String refreshToken = jwtTokenUtil.generateRefreshToken(userDetails.getId(), userDetails.getUsername());
saveRefreshToken(userDetails.getId(), refreshToken, refreshExpiresAt);
return new LoginResponse(accessToken, "Bearer", expiresAt, refreshToken, ..., userInfo);
```

前端在接收到登录响应后，将 Access Token、Refresh Token、用户对象以及 Token 过期时间戳写入 `localStorage` 进行持久化，同时将 Token 写入 Pinia 的 `authStore` 中维护全局登录状态。后续访问受保护页面时，前端路由守卫会检查 `authStore.isAuthenticated`，若本地不存在有效 Token 则自动跳转到登录页。后端则通过 `JwtAuthenticationFilter` 统一校验请求头中 `Authorization` 字段的 Bearer Token，对令牌格式、签名和有效期进行检查，校验通过后将用户身份写入 `SecurityContextHolder`，供后续业务接口直接使用。未携带有效 Token 的请求将被 `RestAuthenticationEntryPoint` 拦截并返回 401 未认证响应。这样处理以后，未登录用户无法直接访问受保护资源，系统的访问控制更加完整。

与面向公众的互联网应用不同，本系统作为企业内部的工单审批类 OA 系统，不提供用户自行注册功能。其原因在于：首先，系统的使用者必须是企业内部经过实名认证的在职员工，账号需要与组织架构中的部门、岗位和汇报关系准确绑定，这些信息无法通过用户自主填写来保证真实性；其次，账号的创建、角色分配和权限授予属于企业人力资源管理或信息化管理的范畴，通常由管理员在员工入职时统一开通，并在调岗或离职时及时调整或回收，自行注册会破坏这种集中管控机制；再次，开放注册会引入安全风险，无法杜绝外部人员或已离职员工获取系统访问权限的可能，而审批系统中往往涉及企业内部的敏感业务流程和财务数据。因此，本系统采用管理员集中开户模式，不提供公开注册入口，这也是同类 OA、ERP、CRM 等企业级管理信息系统的通行做法。

## 5.3 权限管理功能实现

### 5.3.1 用户认证管理

系统的用户认证管理围绕 JWT（JSON Web Token）机制展开，采用双 Token 策略实现身份校验与会话续期。Access Token 有效期为 30 分钟，用于日常接口访问的身份标识；Refresh Token 有效期为 7 天，存储于数据库的 `refresh_token` 表中，用于在 Access Token 过期后换取新的 Token 对。前端在接收到登录响应后，将两枚 Token 与过期时间写入 `localStorage` 进行持久化，后续每次发起 API 请求时，Axios 请求拦截器自动从 `localStorage` 读取当前 Token 并附加到请求头的 `Authorization` 字段中，以 `Bearer` 形式传递。

后端通过 `JwtAuthenticationFilter` 在过滤器链中统一校验请求所携带的令牌。该过滤器继承自 `OncePerRequestFilter`，对每个请求只执行一次校验流程，校验通过后将用户身份写入 `SecurityContextHolder`：

```java
String jwt = getJwtFromRequest(request);
if (StringUtils.hasText(jwt)) {
    String validationError = jwtTokenUtil.validateTokenWithReason(jwt);
    if (validationError == null) {
        Long userId = jwtTokenUtil.getUserIdFromToken(jwt);
        String username = jwtTokenUtil.getUsernameFromToken(jwt);
        UserDetails userDetails = userDetailsService.loadUserByUsername(username);
        UsernamePasswordAuthenticationToken authentication =
                new UsernamePasswordAuthenticationToken(userDetails, null, userDetails.getAuthorities());
        SecurityContextHolder.getContext().setAuthentication(authentication);
    } else {
        request.setAttribute("token_error", validationError);
    }
}
filterChain.doFilter(request, response);
```

`SecurityConfig` 配置类关闭了 CSRF 防护，采用无状态会话策略，所有请求默认需要认证，仅 `/auth/**` 路径允许匿名访问。若校验失败，过滤器在请求属性中设置具体的错误原因（如 `token_expired`、`token_invalid`），由 `RestAuthenticationEntryPoint` 统一返回 401 响应。

为了提升用户体验，前端实现了 Token 的静默刷新机制。Axios 请求拦截器会预检 Token 过期时间，若发现 Access Token 将在 5 分钟内过期且用户在最近 30 分钟内有活动，则提前调用 `POST /auth/refresh` 接口换取新 Token：

```javascript
apiClient.interceptors.request.use(async (config) => {
  const token = localStorage.getItem('token')
  const expiresAt = Number(localStorage.getItem('tokenExpiresAt') || 0)
  const lastActivity = Number(localStorage.getItem('lastActivity') || 0)
  const now = Date.now()
  if (token && expiresAt - now < 5 * 60 * 1000 && now - lastActivity < 30 * 60 * 1000) {
    try { await handleTokenRefresh() } catch (e) { /* 静默失败，由响应拦截器兜底 */ }
  }
  const currentToken = localStorage.getItem('token')
  if (currentToken) config.headers.Authorization = `Bearer ${currentToken}`
  return config
})
```

若后端返回 401 或业务码 401，响应拦截器会尝试使用 Refresh Token 再次刷新并重试原请求；若刷新失败，则清理本地存储并弹出认证过期提示框，引导用户重新登录。登出时，前端调用 `POST /auth/logout` 接口，后端将当前用户关联的所有 Refresh Token 标记为已撤销，前端随后清理所有本地认证数据，确保双端会话同步失效。

### 5.3.2 角色权限管理

角色权限管理功能用于定义系统中的角色及其对应的操作权限集合，并将角色分配给用户。前端管理界面位于 `RoleManage.vue`，管理员可以查看角色列表、创建新角色、编辑角色权限以及删除角色。每个角色包含角色标识、角色名称和权限列表，权限以字符串数组的形式呈现，如 `user_view`、`user_manage`、`approval`、`apply`、`form_design`、`role_manage` 以及通配权限 `all`。

后端 `RoleController` 提供 `/roles` 路径的 RESTful 接口，`RoleServiceImpl` 在创建和更新角色时，使用 FastJSON2 将权限字符串列表序列化为 JSON 字符串后存入 `permissions` 字段；删除与更新操作中对预设的 `admin` 角色进行硬性保护：

```java
@Override
public Long createRole(RoleCreateRequest request) {
    Role exist = roleMapper.selectOne(
            new LambdaQueryWrapper<Role>().eq(Role::getName, request.getName()));
    if (exist != null) throw new BusinessException("角色标识已存在");
    Role role = new Role();
    BeanUtils.copyProperties(request, role);
    role.setPermissions(request.getPermissions() != null
            ? JSON.toJSONString(request.getPermissions()) : "[]");
    roleMapper.insert(role);
    return role.getId();
}
```

角色接口通过 `@PreAuthorize("hasAnyAuthority('role_manage', 'all')")` 限制仅拥有角色管理权限的用户可访问。用户与角色的绑定通过 `User` 实体的 `roleId` 字段实现，前端登录后将用户权限列表缓存于 Pinia Store 和 `localStorage` 中，用于控制页面按钮的显隐和路由访问权限。Vue Router 在 `meta.permissionCheck` 中通过函数式声明对每条受控路由进行权限校验，未通过则重定向到首页。

### 5.3.3 数据权限校验

系统当前实现了基于角色和接口的权限控制，并在工单列表和详情查询中引入了初步的数据级权限隔离。`ApprovalServiceImpl.checkViewPermission` 方法负责校验当前用户是否有权查看指定工单详情：系统管理员可以查看全部工单；部门经理可以查看本部门员工发起的工单以及指派给自己的工单；普通员工只能查看自己发起的工单。工单列表查询接口同样通过 `applyDataPermission` 方法在 `LambdaQueryWrapper` 上动态追加过滤条件：

```java
private void applyDataPermission(LambdaQueryWrapper<Approval> wrapper, User currentUser, Long currentUserId) {
    if ("admin".equals(currentUser.getRoleName())) {
        return; // 管理员无需附加过滤条件
    }
    if ("manager".equals(currentUser.getRoleName())) {
        wrapper.and(w -> w
                .apply("applicant_id IN (SELECT id FROM sys_user WHERE dept_id = {0})",
                       currentUser.getDeptId())
                .or().eq(Approval::getCurrentApproverId, currentUserId));
    } else {
        wrapper.eq(Approval::getApplicantId, currentUserId);
    }
}
```

这种数据权限的实现方式直接嵌入在业务服务层中，通过判断当前用户的角色名称来决定查询范围。虽然当前尚未抽象出独立的数据权限中间件或注解驱动机制，但已能够满足毕业设计场景下的基本需求。更细粒度的数据权限控制（如基于数据行或字段级别的隔离）可作为后续扩展方向，设计上可考虑引入自定义注解和 AOP 拦截器来统一处理。

## 5.4 工单流转功能实现

### 5.4.1 工单创建管理

工单创建管理功能支持用户填写申请表单并保存为草稿或直接提交审批。前端发起审批页面位于 `ApprovalCreate.vue`，页面采用左右分栏布局，左侧为表单填写区，右侧为审批流程提示和辅助信息。用户在创建模式下需选择审批类型（如请假、报销、采购、加班、出差），审批类型下拉框的数据来自 `GET /form-templates/all` 接口返回的启用的表单模板列表。选择审批类型后，前端自动调用 `GET /form-templates/code/{code}` 获取对应模板的字段配置，并渲染 `DynamicForm` 动态表单组件供用户填写扩展信息。页面同时提供审批人预览功能，通过调用 `POST /approver-rules/preview` 接口，系统根据当前用户和审批类型自动解析并展示预计的审批人姓名。

后端工单创建接口为 `POST /approvals`，`ApprovalServiceImpl.create` 在写入数据库前会将动态表单数据使用 FastJSON2 序列化为 JSON 字符串存入 `formData` 字段；对于审批人的确定，系统优先采用前端指定的审批人，若未指定则调用 `DefaultApproverResolver` 自动解析：

```java
Approval approval = new Approval();
BeanUtils.copyProperties(request, approval);
approval.setStatus(ApprovalStatus.DRAFT.getCode());
if (request.getFormData() != null) {
    approval.setFormData(JSON.toJSONString(request.getFormData()));
}
if (request.getCurrentApproverId() != null) {
    validateApproverPermission(request.getCurrentApproverId());
    approval.setCurrentApproverId(request.getCurrentApproverId());
} else {
    ResolverResult result = defaultApproverResolver.resolve(applicantId, request.getType());
    if (result.isSuccess()) approval.setCurrentApproverId(result.getApproverId());
}
approvalMapper.insert(approval);
```

创建成功后工单状态为草稿（`DRAFT`）。编辑工单接口为 `POST /approvals/{id}/update`，仅允许草稿状态的工单进行编辑，编辑时同样支持重新指定或自动解析审批人。保存草稿功能实际上是调用创建接口，仅将数据持久化而不触发状态流转。

### 5.4.2 审批状态流转

审批状态流转是系统的核心功能，采用阿里巴巴 COLA 状态机引擎实现。状态机配置位于 `StateMachineConfig`，定义了名为 `approvalStateMachine` 的状态机实例。每条状态转换规则包含源状态、目标状态、触发事件、条件（Condition）和动作（Action）五要素，通过链式 API 声明：

```java
StateMachineBuilder<ApprovalStatus, ApprovalEvent, ApprovalContext> builder =
        StateMachineBuilderFactory.create();

builder.externalTransition()
        .from(ApprovalStatus.DRAFT).to(ApprovalStatus.PROCESSING).on(ApprovalEvent.SUBMIT)
        .when(stateMachineHelper::checkFormComplete)
        .perform(stateMachineHelper::doSubmit);

builder.externalTransition()
        .from(ApprovalStatus.PROCESSING).to(ApprovalStatus.APPROVED).on(ApprovalEvent.APPROVE)
        .when(stateMachineHelper::checkApproverPermission)
        .perform(stateMachineHelper::doApprove);

builder.externalTransition()
        .from(ApprovalStatus.PROCESSING).to(ApprovalStatus.RETURNED).on(ApprovalEvent.REJECT)
        .when(stateMachineHelper::checkApproverPermission)
        .perform(stateMachineHelper::doReject);

// 撤销与重新编辑两条转换同理省略
StateMachine<ApprovalStatus, ApprovalEvent, ApprovalContext> stateMachine = builder.build(STATE_MACHINE_ID);
```

条件检查和动作执行逻辑封装在 `ApprovalStateMachineHelper` 中：提交操作的条件 `checkFormComplete` 校验工单的 `formData` 字段是否非空；审批通过和拒绝的条件 `checkApproverPermission` 综合校验当前用户是否为指定审批人、管理员代审批人或部门经理代审批人；撤销和重新编辑的条件 `checkIsApplicant` 校验当前用户是否为工单申请人。状态机的触发统一在 `ApprovalServiceImpl` 中通过 `stateMachine.fireEvent(currentStatus, event, context)` 执行，若返回的状态与当前状态相同则视为转换失败，抛出业务异常提示用户当前操作不被允许。

### 5.4.3 审批操作管理

审批操作管理功能提供审批通过、审批拒绝、重新编辑和撤销申请四种核心操作。前端审批详情页 `ApprovalDetail.vue` 根据当前工单状态和当前用户身份动态展示可操作按钮。当工单处于审批中状态且当前用户具备审批执行权限时，页面显示审批意见输入框、"通过审批"和"拒绝审批"按钮；若当前用户是申请人但非审批人，则仅显示"撤销申请"按钮；当工单为草稿且当前用户是申请人时，显示"提交审批"和"编辑内容"按钮；当工单为已通过或已打回且当前用户是申请人时，显示"重新编辑"按钮。

后端接口分别为 `POST /approvals/{id}/approve`、`POST /approvals/{id}/reject`、`POST /approvals/{id}/reedit` 和 `POST /approvals/{id}/revoke`。系统实现了四级审批权限模型，由 `ApprovalStateMachineHelper.checkApproverPermissionDetail` 方法集中判定：

```java
// Level 1: 指定审批人直接审批
if (currentUserId.equals(approverId)) {
    if (!hasPermission(currentUser, PERMISSION_APPROVAL_EXECUTE))
        return ApprovalPermissionResult.denied("您没有审批权限，请联系管理员");
    return ApprovalPermissionResult.granted(ApprovalActionType.DIRECT, "直接审批");
}
// Level 2: 管理员全范围代审批
if (ROLE_ADMIN.equals(roleName) && hasPermission(currentUser, PERMISSION_APPROVAL_EXECUTE_ALL)) {
    return ApprovalPermissionResult.granted(
            ApprovalActionType.PROXY_ADMIN, "管理员代审批", approverId);
}
// Level 3: 部门经理本部门代审批
if (ROLE_MANAGER.equals(roleName) && hasPermission(currentUser, PERMISSION_APPROVAL_EXECUTE_DEPT)) {
    if (isSameDepartment(approval.getApplicantId(), currentUserId)) {
        return ApprovalPermissionResult.granted(
                ApprovalActionType.PROXY_MANAGER, "部门经理代审批", approverId);
    }
    return ApprovalPermissionResult.denied("您只能审批本部门的工单");
}
// Level 4: 无权限
return ApprovalPermissionResult.denied("无权执行审批操作，您不是当前审批人");
```

第一级为指定审批人直接审批；第二级为系统管理员全范围代审批，要求拥有 `approval:execute:all` 权限；第三级为部门经理本部门代审批，要求拥有 `approval:execute:dept` 权限且申请人与经理属于同一部门；第四级为无权限拒绝。权限检查结果 `ApprovalPermissionResult` 被放入状态机上下文，供动作方法记录代审批信息到审批历史表中。审批操作接口同时使用 `@PreAuthorize("hasAnyAuthority('approval', 'approval:execute', 'all')")` 进行接口级别的权限控制。

### 5.4.4 审批历史管理

审批历史管理功能记录工单全生命周期中的每一次状态变更和操作行为，为审计追溯提供依据。后端 `ApprovalHistory` 实体对应数据库表，字段包括工单 ID、操作人 ID、操作类型、审批意见、创建时间，以及扩展的代审批信息字段（是否代审批、代审批类型、原审批人 ID、代审批原因）。

每当状态机的动作方法被触发时，`ApprovalStateMachineHelper.saveHistory` 方法被调用以持久化操作记录。对于代审批场景，方法会从 `ApprovalPermissionResult` 中提取原审批人 ID 与代审批类型一并落库：

```java
ApprovalHistory history = new ApprovalHistory();
history.setApprovalId(approvalId);
history.setApproverId(approverId);
history.setAction(action);
history.setComment(comment);
history.setCreateTime(LocalDateTime.now());
if (permissionResult != null && permissionResult.isProxyApproval()) {
    history.setApprovalType(permissionResult.getApprovalTypeCode());
    history.setIsProxy(1);
    history.setOriginalApproverId(permissionResult.getOriginalApproverId());
    history.setProxyReason(permissionResult.getApprovalTypeLabel());
} else {
    history.setApprovalType(ApprovalActionType.DIRECT.getCode());
    history.setIsProxy(0);
}
approvalHistoryMapper.insert(history);
```

正常审批操作类型记录为 `DIRECT`；代审批根据权限结果记录为 `PROXY_ADMIN` 或 `PROXY_MANAGER`，审批意见在代审批场景下会自动追加前缀标识，如"[管理员代审批] 同意"。后端提供 `GET /approvals/{id}/history` 接口返回包含操作人姓名、操作类型名称、审批意见和时间戳的完整记录。前端 `ApprovalDetail.vue` 在页面加载时调用该接口，以时间轴形式展示审批历史列表，便于申请人和管理人员追踪工单处理过程。

### 5.4.5 待办已办管理

待办已办管理功能为审批人提供任务视角的工单查询。待办列表展示所有处于审批中状态且当前用户有权处理的工单，已办列表展示当前用户曾经处理过的工单。前端待办页面 `TodoList.vue` 和已办页面 `DoneList.vue` 分别通过调用 `approvalStore.fetchTodoList()` 和 `approvalStore.fetchDoneList()` 加载数据，并以卡片或表格形式展示工单标题、申请人、部门、优先级和创建时间。

后端 `ApprovalServiceImpl.getTodoList` 在查询条件中根据当前用户的角色区分管理员视图与普通视图：

```java
LambdaQueryWrapper<Approval> wrapper = Wrappers.lambdaQuery();
wrapper.eq(Approval::getStatus, ApprovalStatus.PROCESSING.getCode());
if (currentUser == null || !"admin".equals(currentUser.getRoleName())) {
    wrapper.eq(Approval::getCurrentApproverId, approverId);
}
wrapper.orderByDesc(Approval::getCreateTime);
List<Approval> approvals = approvalMapper.selectList(wrapper);
```

已办接口 `GET /approvals/done` 的实现方式则是先从 `approval_history` 表中查询当前用户处理过的所有历史记录，提取去重后的工单 ID 集合，再反查 `approval` 表获取工单的最新详情。这种基于历史表反查的设计能够准确反映用户参与过的审批活动，即使工单后续状态已发生变化，依然能在已办列表中呈现。两个接口均返回分页结果，前端使用独立的分页状态存储待办总数和已办总数，用于侧边栏徽章展示。

## 5.5 表单模板管理功能实现

### 5.5.1 表单模板管理

表单模板管理功能允许管理员预定义各类审批业务所需的表单结构，从而在不修改源代码的前提下扩展系统所支持的审批类型。前端表单设计器位于 `FormDesigner.vue` 页面，通过响应式变量 `mode` 在列表模式与设计模式之间切换。列表模式展示所有表单模板的名称、编码、描述和启用状态，支持新建、编辑和删除操作；设计模式采用三栏布局，左侧为组件库，中间为画布区，右侧为表单属性设置区。

组件库提供八种基础表单组件，覆盖了企业审批表单中绝大多数字段类型的录入需求。组件类型集合在脚本入口处以静态数组形式声明：

```javascript
const formComponents = [
  { type: 'text', label: '单行文本' },
  { type: 'textarea', label: '多行文本' },
  { type: 'number', label: '数字输入' },
  { type: 'email', label: '邮箱输入' },
  { type: 'date', label: '日期选择' },
  { type: 'select', label: '下拉选择' },
  { type: 'radio', label: '单选框' },
  { type: 'checkbox', label: '多选框' }
]
```

管理员点击组件库中的某项条目时，`addComponent` 方法将该组件追加到 `formFields` 响应式数组的末尾，并以当前时间戳生成唯一标识，从而触发画布区的重新渲染：

```javascript
function addComponent(component) {
  formFields.value.push({
    id: 'field_' + Date.now(),
    type: component.type,
    label: component.label,
    name: `field_${Date.now()}`,
    placeholder: '',
    required: false,
    optionsText: ''
  })
}
```

画布中的每个字段卡片支持就地编辑字段名称、字段标识、占位提示、是否必填以及选项内容（针对选择类组件）；通过 `moveField` 与 `removeField` 方法分别实现字段上下移动和删除操作，配合 Vue 的响应式机制实时更新视图。当管理员点击"保存表单"按钮时，前端调用 `buildSaveData` 方法将画布状态归一化为后端可接受的请求体：选项类组件的多行文本被拆分为标签—值对的对象数组，非选项类字段则只保留基础属性。

```javascript
function buildSaveData() {
  const fieldsConfig = formFields.value.map(f => {
    const field = {
      id: f.id, type: f.type, label: f.label, name: f.name,
      placeholder: f.placeholder, required: !!f.required
    }
    if (f.type === 'select' || f.type === 'radio' || f.type === 'checkbox') {
      field.options = f.optionsText.split('\n').filter(Boolean)
              .map(s => ({ value: s, label: s }))
    }
    return field
  })
  return { name: formSettings.value.name, code: formSettings.value.code,
           description: formSettings.value.description,
           fieldsConfig, status: formSettings.value.status }
}
```

后端表单模板接口由 `FormTemplateController` 提供，基础路径为 `/form-templates`，包括查询全部启用的模板 `GET /form-templates/all`、分页查询 `GET /form-templates`、按编码查询 `GET /form-templates/code/{code}`、创建 `POST /form-templates`、更新 `POST /form-templates/{id}/update` 和删除 `POST /form-templates/{id}/delete`。其中按编码查询接口主要服务于工单创建场景，前端在用户选择审批类型后立即调用该接口拉取对应模板的字段配置以驱动动态表单渲染。`FormTemplateServiceImpl` 在创建表单模板时，先通过 `LambdaQueryWrapper` 校验编码的唯一性，再使用 FastJSON2 将前端的字段配置数组序列化为 JSON 字符串存入 `fieldsConfig` 字段，从而以单字段方式承载结构化的表单定义：

```java
@Override
public Long createTemplate(FormTemplateCreateRequest request) {
    FormTemplate exist = formTemplateMapper.selectOne(
            new LambdaQueryWrapper<FormTemplate>().eq(FormTemplate::getCode, request.getCode()));
    if (exist != null) {
        throw new BusinessException("表单编码已存在");
    }
    FormTemplate template = new FormTemplate();
    BeanUtils.copyProperties(request, template);
    if (request.getFieldsConfig() != null) {
        template.setFieldsConfig(JSON.toJSONString(request.getFieldsConfig()));
    } else {
        template.setFieldsConfig("[]");
    }
    if (template.getStatus() == null) {
        template.setStatus(1);
    }
    template.setCreateTime(LocalDateTime.now());
    formTemplateMapper.insert(template);
    return template.getId();
}
```

更新方法 `updateTemplate` 沿用相同的序列化策略，但不允许修改表单编码，避免编码变更导致的关联失效。表单编码用于在工单创建时与审批工单类型建立关联。整个表单模板模块的写操作接口（创建、更新、删除）均通过 `@PreAuthorize("hasAnyAuthority('form_design', 'all')")` 限制仅拥有表单设计权限的用户可操作，而读操作则放宽为 `isAuthenticated()`，使所有已登录用户都能在发起审批时获取所需的模板结构。这种读写分离的权限设计在保证模板维护安全性的同时，兼顾了表单填写场景下的可访问性。

### 5.5.2 动态表单管理

审批规则的配置管理功能为管理员提供了规则的增删改查能力。前端管理界面位于ApproverRuleManage.vue，规则列表页展示规则名称、匹配策略类型、审批人类型、优先级和状态，支持按关键词搜索和按策略类型、状态筛选。管理员可以创建新规则，配置规则名称、匹配条件（部门、审批类型、角色）、解析策略、审批人值、优先级和描述信息；也可以编辑已有规则或删除不再使用的规则。

后端ApproverRuleController提供 /approver-rules路径的接口，包括创建POST /approver-rules、更新PUT /approver-rules/{id}、删除DELETE /approver-rules/{id}、查询详情GET /approver-rules/{id}、分页查询GET /approver-rules和效果预览POST /approver-rules/preview。ApproverRuleServiceImpl在创建和更新时，将匹配条件对象序列化为JSON字符串存入matchConditions字段。分页查询支持按关键词、策略类型和状态筛选，并支持按优先级或名称排序。规则配置接口通过 @PreAuthorize("hasAnyAuthority('role_manage', 'all')") 限制仅角色管理员可访问。效果预览接口则对所有已登录用户开放，便于普通用户在发起审批前了解审批人的分配逻辑。
