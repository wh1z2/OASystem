### 2.1 用户管理接口 (`UserController`)

| 方法 | 路径 | 权限要求 | 说明 |
|------|------|----------|------|
| GET | `/users` | `user_view` / `user_manage` / `all` | 分页查询用户列表 |
| GET | `/users/{id}` | `user_view` / `user_manage` / `all` | 根据ID查询用户详情 |
| POST | `/users` | `user_manage` / `all` | 创建用户 |
| POST | `/users/{id}/update` | `user_manage` / `all` | 更新用户 |
| POST | `/users/{id}/delete` | `user_manage` / `all` | 删除用户 |
| POST | `/users/password` | 已认证即可 | 修改当前用户密码 |
| POST | `/users/profile` | 已认证即可 | 更新当前用户资料 |

## 5. 已知限制与说明

1. **部门字段**：前端用户表单中"部门"为文本输入，但后端 `UserCreateRequest` / `UserUpdateRequest` 仅接受 `deptId`（Long 类型）。由于当前系统暂无部门列表查询接口，创建/更新用户时不传递 `deptId`，部门信息由后端根据已有 `deptId` 关联查询后返回。
2. **角色映射**：前端使用角色名称字符串（如 `admin`），后端使用 `roleId`。前端 `stores/user.js` 通过已加载的 `roles` 列表自动完成名称到 ID 的映射，因此打开用户管理页面前需确保 `fetchRoles()` 已成功执行。
3. **状态转换**：后端用户状态使用整数（`0`=禁用，`1`=启用），前端展示时映射为字符串（`inactive` / `active`）。编辑用户时暂未开放状态切换 UI，如需禁用用户需直接操作数据库或扩展接口。
