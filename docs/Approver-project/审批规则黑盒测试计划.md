# 审批规则管理黑盒测试计划

## 1. 测试概述

本测试计划从最终用户视角出发，通过前端页面交互和实际系统表现，验证审批规则管理模块及自动审批人解析功能是否符合预期。

**测试环境要求**：
- 前端服务正常运行（`npm run dev`，默认端口 3000）
- 后端服务正常启动且 Flyway 迁移已执行（含 `V1_3__Fix_Approver_Rule_Data.sql`）
- 浏览器：Chrome / Edge 最新稳定版
- 测试账号：admin / admin123（系统管理员，系统管理部）

## 6. 边界与异常场景测试

| 用例ID | 场景描述 | 操作步骤 | 预期结果 |
|---|---|---|---|
| EDGE-001 | 申请人未分配部门 | 在数据库中将某用户 dept_id 设为 null，用该用户登录创建请假单 | 预览区域或提交时提示「未分配部门」或「无法定位部门负责人」 |
| EDGE-002 | 申请人所在部门无部门经理 | 在数据库中删除某部门下所有 role_id=2 的用户，该部门员工创建工单 | 提示「无法定位部门负责人」 |
| EDGE-003 | 禁用规则不参与匹配 | 将「技术部请假规则」状态改为禁用，技术部员工创建请假单 | 不再命中该禁用规则；若存在其他匹配规则则走其他规则，否则走兜底策略 |
| EDGE-004 | 规则优先级冲突 | 创建两条规则：A 匹配所有报销、指定 admin（优先级 1）；B 匹配所有报销、指定张经理（优先级 100）。李四创建报销单 | 系统优先命中规则 A，审批人为 admin |
| EDGE-005 | 自审防护 | 创建规则：系统管理部请假 → 指定用户 → 系统管理员自己。系统管理员创建请假单 | 系统跳过该规则（不自审），继续匹配下一条可用规则，最终返回赵经理 |
| EDGE-006 | 规则匹配条件为空（全局兜底规则） | 创建一条 matchConditions 为空的规则，指定审批人为张经理 | 任何用户创建任何类型工单都优先命中该规则，审批人为张经理 |
| EDGE-007 | 审批人指定的用户已被禁用 | 创建规则指定某用户为审批人，随后将该用户在用户管理中禁用。匹配该规则创建工单 | 该规则解析失败，继续匹配下一条规则 |
| EDGE-008 | 审批人指定的角色在部门中无人担任 | 创建规则：某部门 → 指定角色 → 管理员（该部门无管理员角色用户） | 该规则解析失败，继续匹配或走兜底策略 |

## 9. 测试通过标准

1. **审批规则管理页面**：所有规则的审批人列均显示正常中文名称，无 "角色#X"、"用户#X" 等未解析文本。
2. **自动解析功能**：各测试账号按上表场景创建工单时，系统能正确解析出预期审批人。
3. **手动指定优先**：手动指定审批人时，系统严格遵守用户选择，不走自动解析。
4. **兜底策略**：无匹配规则时，系统能正确找到申请人所在部门的部门经理。
5. **异常处理**：自审、禁用规则、无效用户、无部门经理等边界场景均有合理提示，系统不崩溃。
6. **接口稳定性**：整个测试过程中，相关后端接口无 500 内部错误。
