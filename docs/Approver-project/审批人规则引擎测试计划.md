# 审批人规则引擎测试计划

## 1. 测试目标

验证审批规则管理模块的完整功能链路，确保：
1. 审批规则的增删改查（CRUD）功能正确。
2. 审批人解析引擎（`DefaultApproverResolver`）能根据申请人、审批类型、部门等条件正确匹配规则并解析出审批人。
3. 自动解析与手动指定审批人的集成逻辑无误，不影响原有功能。
4. 数据修复（如角色ID误存为用户ID）后，前端展示与后端解析均恢复正常。

## 2. 测试范围

| 层级 | 测试对象 | 说明 |
|---|---|---|
| 单元测试 | `ApproverRuleService` | 规则的 CRUD、分页查询、预览接口 |
| 单元测试 | `DefaultApproverResolver` | 策略匹配、审批人解析、兜底策略、自审跳过 |
| 集成/回归测试 | `ApprovalService` | 创建/提交工单时自动解析审批人的集成效果 |
| 数据层 | Flyway 迁移脚本 | 数据修复脚本（`V1_3__Fix_Approver_Rule_Data.sql`）的执行与效果 |

## 4. 测试策略

采用 **@SpringBootTest + @Transactional** 的自动化测试策略：
- 所有测试在事务中执行，测试结束后自动回滚，保证数据隔离。
- 解析引擎测试直接操作 `approver_rule` 表插入临时规则，验证匹配逻辑。
- 回归测试通过 `ApprovalService` 的 `create`/`submit` 接口验证端到端流程。

## 7. 边界与异常测试

| 场景 | 预期行为 |
|---|---|
| `matchConditions` 中 `deptIds` 不包含申请人部门 | 规则不匹配，继续下一条 |
| `matchConditions` 中 `types` 不包含当前审批类型 | 规则不匹配，继续下一条 |
| `matchConditions` 中 `roleIds` 不包含申请人角色 | 规则不匹配，继续下一条 |
| `approverValue` 为非法 JSON | `JSON.parseArray` 异常，视为解析失败，继续下一条 |
| `approverValue` 为空数组 `[]` | 解析失败，继续下一条 |
| 申请人 `dept_id` 为 null | 兜底策略直接失败，提示未分配部门 |
| 申请人所在部门无部门经理（`role_id=2` 且状态=1 的用户） | 兜底策略失败，提示无法定位部门负责人 |

## 10. 风险与注意事项

1. **数据基线漂移**：`sys_user` 中用户的部门、角色若发生变更，需同步更新测试用例中的期望值。
2. **JSON 序列化兼容性**：`matchConditions` 中 `types` 字段存储格式（字符串数组 vs 数字数组）直接影响 `typeMatches` 的匹配结果。未来若统一为字符串编码，需同步清理数据库历史数据。
3. **规则优先级数值**：优先级数值越小表示优先级越高。插入测试规则时需注意数值设计，避免意外覆盖生产规则。
4. **Flyway 迁移顺序**：`V1_3__Fix_Approver_Rule_Data.sql` 必须在应用启动前执行，否则修复后的测试用例会失败。
